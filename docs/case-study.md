# Case study: building PitCrew

## The problem

The starting point was a small, working FastAPI service: two scikit-learn models predicting when a vehicle needs its next service, plus a rules-based fallback, no auth, no database, JSON files for storage.
It was a reasonable weekend project, but it did not read as senior AI/LLM-engineer work: no agent orchestration, no RAG, no evals, no observability, no real persistence, no auth, and (as investigation in Phase 3 found) a genuine, silent bug in the training data that made every "fresh" vehicle predict a near-zero remaining service window regardless of its actual history.

The goal for this rebuild was explicit from the start: turn that service into a full-stack AI product that demonstrates the specific things an AI/LLM-engineering hiring manager screens for - multi-agent orchestration, a real RAG pipeline with citations, evaluations wired into CI as a regression gate, and traced observability - while staying honest about what is and is not actually finished.

## The design

The rebuild kept the original 3-tier prediction cascade (model v2, model v1, deterministic rules) as one tool a new multi-agent assistant can call, rather than replacing it.
Around that core, four major subsystems were added in sequence, each building on the last: a real async Postgres persistence layer with Argon2id auth and RBAC (Phases 1-2); a corrected ML predictor behind an explicit model registry (Phase 3); a Next.js dashboard (Phase 4); a LangGraph supervisor routing to PydanticAI specialists behind a pluggable `LLMClient` (Phase 5); hybrid RAG with citations (Phase 6); deterministic replay-mode cassettes so CI never depends on a live LLM (Phase 7); a streaming assistant UI (Phase 8); and CI, evals, and deploy configuration (Phase 9).

Two decisions shaped almost everything downstream.
First, the hard "no paid LLM API, ever" constraint pushed toward Groq's free tier for live chat completions and local CPU models for embeddings and reranking - a real product would likely make a different call here, but a portfolio project that can never cost its author money was a deliberate design goal, not an accident (see ADR 0005).
Second, choosing LangGraph over a simpler hand-written router was an explicit tradeoff, not an obvious win: the spec itself names the tension (a hand-written supervisor would have been easier to read) and accepts the extra framework dependency because LangGraph is a stronger, more recognizable signal to reviewers (see ADR 0002).

The RAG pipeline (ADR 0003) was built to genuinely need both of its two retrieval signals: a deliberately-imperfect eval question set (Phase 9) proves this by including a case where an irrelevant Honda Civic passage outranks the correct BMW passage before reranking, and a case where the generated answer contains one unsupported claim the retrieved context does not back - both cases score meaningfully below 1.0, confirming the eval suite discriminates rather than trivially passing everything.

## Tradeoffs and what's actually still open

This project deliberately did not chase a flawless finish line.
The following are real, disclosed gaps, most surfaced during this project's own phase-by-phase code review (`.superpowers/sdd/progress.md`), not swept under the rug for a cleaner narrative:

- **The live deploy has not happened.** No Neon, Render, Vercel, or Langfuse Cloud account exists as of this writing. Phase 9 built and locally verified all of the deploy configuration (including a real `docker build`/`docker run`/`curl /health` cycle) but never provisioned a live account. This is the single biggest remaining gap between "documented as production-ready" and "actually running in production" - see the README's [deployment checklist](README.md#deployment-checklist) via the repo root README.
- **The Docker image is heavier than it should be.** It pulls in `torch`'s default CUDA/GPU wheel via `sentence-transformers`, even though this project's design is explicitly CPU-only everywhere. It may not fit Render's free-tier build/memory limits until `requirements.txt` pins `torch`'s CPU-only wheel index - flagged in Phase 9, not yet fixed.
- **Scheduling writes to memory, not a table.** `InMemoryScheduler` is a reasonable stand-in for a demo (the confirm-before-write guardrail is real and tested against the real tool boundary), but a booking does not survive a restart. A real `appointments` table is the natural next step whenever scheduling needs to be durable.
- **The SSE "streaming" answer is word-chunked after the fact, not true token streaming.** No `LLMClient` implementation in this codebase streams incrementally yet - both `GroqClient` and `ReplayClient` return one complete response per call. The agent-activity `trace` events in the same stream are genuinely real-time per-LangGraph-node output; only the final answer text is simulated as a stream. Any external-facing copy about this feature should say "streams a live trace, with the answer revealed progressively" rather than imply token-level model streaming.
- **A citation-replay fragility exists and is understood, not fixed.** Deterministic cassette replay for knowledge questions depends on `KBChunk.chunk_id` being a stable autoincrement primary key, which only holds because the test suite resets the schema on every run. The failure mode is safe (a shifted id causes a loud cassette miss, never a silently wrong citation), and fixing it properly (hashing chunk content instead of a DB id) would touch the shipped citation schema, so it was reasonably deferred rather than rushed.
- **The RAG refusal threshold is calibrated, not independently validated.** The one test asserting "the Knowledge agent refuses when retrieval is weak" reuses one of the same 15 probe questions used to calibrate that threshold in the first place. The mechanism is real and exercises the real pipeline, but it has not been proven to generalize to genuinely unseen questions - a larger, independently-held-out eval set is the natural fix.
- **The eval suite's scores are real, but replay-mode.** `make eval`'s Ragas suite (faithfulness/context precision/context recall/answer relevancy means: 0.9167 / 0.8333 / 0.9167 / 0.9319) and DeepEval suite (trajectory and tool-selection both 1.0) both genuinely execute the real scoring code against a real, small (3-case) question set, judged via hand-synthesized cassettes under `LLM_BACKEND=replay`, reproduced byte-for-byte across multiple runs - not placeholder numbers. They are not, however, a live-Groq-judged run, and the question set is intentionally small (enough to prove the pipeline discriminates, not enough to be a comprehensive quality bar). A future manual run against a live judge, on a larger question set, is the natural next validation step before treating these thresholds as an ongoing production quality gate rather than a smoke test.
- **A handful of smaller, low-risk gaps remain from earlier phases**, none blocking: no email-format validation on registration; no route yet lets an owner claim a vehicle; the BFF's own session cookie uses `SameSite=lax` rather than `strict` (a deliberately narrower scope than the backend refresh cookie's stricter setting, not a spec violation); a stale docstring reference in `guardrails.py`; a `scikit-learn` version-mismatch warning on model load that has not caused an actual failure. Each is recorded in its originating phase's log entry in `.superpowers/sdd/progress.md`.

## What's next

In rough priority order: provision the real Neon/Render/Vercel/Langfuse Cloud accounts and complete the actual deploy (the biggest gap); pin `torch`'s CPU-only wheel index before that deploy, so the free-tier build has a realistic chance of succeeding; replace the in-memory scheduler with a real `appointments` table; expand the eval question set and run it once against a live Groq judge to sanity-check the replay-mode numbers; and, if the assistant's "streaming" framing ever becomes customer-facing rather than portfolio-facing, either implement genuine token-level streaming in `LLMClient` or make the "trace is real, text is chunked" distinction explicit in the UI copy itself.
